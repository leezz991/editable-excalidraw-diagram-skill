# 中文实现说明

这个 skill 的目标是把 AI 生成的技术路线图、研究框架图、流程图转成可继续编辑的 `.excalidraw` 文件。经过实际测试，最稳的方案不是“整张图作为底图，只叠加文字”，也不是“把所有图标和插画都拆成线条重画”，而是混合方案：

- 结构元素用 Excalidraw 原生元素：文本框、分组框、合并框、标题条、连接线、箭头、虚线框。
- 复杂视觉元素用图片：图标、小图案、场景插画、徽章、装饰符号。
- 文字用 Excalidraw 文本元素，保持可选中、可复制、可修改。

## 为什么采用混合方案

技术路线图中真正需要后续编辑的通常是结构和文字：框的位置、箭头关系、分组范围、标题和说明。图标和插画更多是视觉辅助，如果强行用很多线段拼出来，会带来几个问题：

- 元素数量膨胀，编辑体验差。
- 图标容易失真，尤其是圆形图标、复杂符号、道路/桥梁插画。
- 小图标的线条碎片没有实际编辑价值。
- 复刻成本高，但收益低。

所以默认规则是：结构可编辑，文字可编辑，复杂图标图片化。

## 实现流程

1. 读取源图片和布局 JSON。
2. 根据 `rectangles` 生成 Excalidraw 矩形、标题条、分组框、虚线框等。
3. 根据 `arrows` 生成 Excalidraw 箭头和连接线。
4. 根据 `assets` 裁切源图中的图标/插画，或读取外部透明 PNG，写入 `.excalidraw` 的 `files` 字段。
5. 根据 `texts` 生成可编辑文本元素。
6. 输出 `.excalidraw`、图片资产目录、MCP elements JSON 和文字核对清单。
7. 通过 Excalidraw MCP `create_view` 生成 checkpoint，并可上传成 Excalidraw 链接。

## 关键经验

### 1. 文本框要使用完整布局尺寸

不要根据字符数量估算文字宽度后缩小 text element。中文在 Excalidraw 里的字体度量和本地预览可能不同，过窄会导致显示不全或裁切。实践中应让文本元素直接使用布局 JSON 给定的完整 `width` 和 `height`。

### 2. 小图标不要强行裁源图

源图里的小图标经常贴近文本框边线、浅色底、阴影或装饰线。直接裁切容易带出异常底色，放回 Excalidraw 后会和背景融合不好。

处理顺序：

1. 源图裁切干净时，直接使用 `crop`。
2. 背景是单一颜色时，使用 `transparent: { "mode": "corner", "tolerance": 36 }` 去掉角落背景色。
3. 裁切仍然带脏边时，创建干净透明 PNG，并在 asset 中使用 `source`。

### 3. 结构线框仍然要原生生成

用户希望后续能编辑文本框、合并框、分组框和连接线，所以这些不应作为整张底图的一部分。它们应该是 Excalidraw 原生 rectangle/arrow 元素。

### 4. 文字核对清单必不可少

默认不依赖重型 OCR。Codex 视觉识别后必须生成 `text-audit.md`，让用户能快速检查中文是否漏字、错字、换行错误。

## 适合场景

- AI 生成的中文技术路线图。
- 研究框架图、实施路径图、流程图。
- 含少量图标、插画、场景图的汇报图。
- 用户需要继续编辑框线、箭头和文字，但不要求图标完全矢量可编辑。

## 不适合场景

- 用户要求所有图标也必须拆成完全可编辑的矢量线条。
- 用户要求全自动 OCR 且不接受人工核对。
- 源图极其复杂，图标和结构边界难以分离。
- 用户只想快速改文字、不关心线框编辑，这种情况可改用 legacy cleaned-background 工作流。
